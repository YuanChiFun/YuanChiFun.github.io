MVC,MVVM,MVP

发表于 2019-03-25 | 分类于 -框架
字数统计 950 | 阅读时长 3

我们在使用框架的时候很少去了解它的架构是什么，这些架构是这些框架实现的基础。所以我们来了解一下MV*

MVC

MVC的意思是，软件可以分为三个部分。

  • 视图(View): 用户界面。
  • 控制器(Controller): 业务逻辑
  • 模型(Model):数据保存

各部分之间的通信方式如下:

  • View传送指令到Controller
  • Controller完成业务逻辑后，要求Model改变状态
  • Model将新的数据发送到View，用户得到反馈

所有通信都是单向的

互动模式

MVC互动模式可以分为两种

  • 通过View接收指令，传递给Controller
    • 用户传指令给View
    • View传送指令到Controller
    • Controller完成业务逻辑后，要求Model改变状态
    • Model将新的数据发送到View，用户得到反馈
  • 直接通过Controller接受指令
    • 用户将指令传给Controller
    • Controller完成业务逻辑后，要求Model改变状态
    • Model将新的数据发送到View，用户得到反馈

MVP

MVP模式将Controller改名为Presenter，同时改变了通信方向

  • 各部分之间的通信，都是双向的
  • View与Model不发生联系，都通过Presenter传递。
  • View非常薄，不部署任何业务逻辑，称为“被动视图”，没有任何主动性
  • Presenter非常厚，所有逻辑都部署在那里

MVVM

MVVM模式将Presenter改名为ViewModel，基本上与MVP模式完全一致

区别是，采用双向绑定，View的变动，自然反映在ViewModel。

区别

可以看出，这三个架构区别在M和V的联系。

Controller

Controller：负责监听View的用户事件，得到数据后Controller做一些处理，然后渲染View。

但是随着逻辑的复杂，这样的处理就会遇到很难调试的问题。由于View一定要运行在UI环境下，而且Model和Controller和View强耦合，没有办法单独验证应用逻辑的正确性。当出现了问题之后，因为各个模块是耦合在一起的，也不能快速判断究竟是哪个模块出现的问题。因此，MVP模式出现了

Presenter

Presenter：比起Controller，Presenter会调用View层提供的接口去渲染Model。

这样做的好处有：

  • 面向接口编程
  • 更好的解耦
  • 方便做单元测试

现在P和V解耦了，P可以自己做单元测试了。软件结构划分更加清楚，逻辑清晰并方便调试。这是因为View层提供了接口。当一个UI复杂起来的时候，View层需要提供的接口是很多的，这本身也是一种开发调试的成本。

ViewModel

ViewModel：比起MVP中的View需要自己提供API，MVVM在VM中构建一组状态数据，作为View状态的抽象。然后通过双向数据绑定使VM中的状态数据与View中的显示状态保持一致。这样，VM的展示逻辑只需要修改对应的状态数据，就可以控制View的状态。从而避免在View上开发大量的接口

但是当UI比较简单的时候，使用VM就会使业务逻辑变得复杂，过分设计。

一个简单的栗子

我们在网上购物的时候，如果这个页面看完了，我们下拉刷新物品。三种架构的处理方式分别为

  • View获取下拉事件，通知Controller
  • Controller向后台Model发起请求，请求内容为下拉内容。
  • Model获得物品数据，传递给Controller
  • Controller拿到数据，做一些数据处理，然后渲染View
    • MVC: 拿到UI节点，渲染物品数据
    • MVP：通过View提供的接口渲染数据
    • MVVM：无需操作，只需要VM数据的彼岸花，通过双向数据绑定，View直接变化。·
